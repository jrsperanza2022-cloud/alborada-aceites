CRÍTICO Bug A — calcSaldo(prov/comp): saldo siempre cero

calcSaldo() usaba N('f-tv') para leer el total de venta. N() internamente hace parseFloat(el.value), que sobre el valor "$28.050.000" retorna NaN (el signo pesos y los puntos de miles rompen el parsing). El saldo quedaba vacío aunque hubiera datos.

const tv=N('f-tv'); // FIX: extraer solo dígitos del campo readonly formateado const tv=parseInt((document.getElementById('f-tv')?.value||'').replace(/[^\d]/g,''))||0;
Ubicación: función calcSaldo(), ramas actor==='prov' y actor==='comp'

CRÍTICO Bug B — calcSaldo(ref): saldo incorrecto

Para el refinador, el saldo se calculaba como IVA(costoNeto) - pagado, pero el campo "Pagado ($)" del refinador guarda el importe neto pagado (sin IVA). Esto producía un saldo exagerado en ~21%.

if(c)document.getElementById('f-ref-saldo').value=MF(IVA(c)-p); // FIX: refPagado es neto → restar del costo neto, no del c/IVA if(c)document.getElementById('f-ref-saldo').value=MF(c-p);
Ubicación: función calcSaldo(), rama actor==='ref'

ALTO Bug C — saveOp(): 4 claves duplicadas en objeto data

El objeto data en saveOp() declara provMoneda, refMoneda, transpMoneda y comMoneda dos veces cada una. En JavaScript la segunda declaración pisa a la primera. En este caso ambas leen el mismo campo, por lo que no genera datos incorrectos, pero es una inconsistencia y un bug latente si se cambia una de las dos.

CRÍTICO Bug D — saveOp() no guarda tnVenta; editOp() la pierde

editOp() recupera las toneladas de venta con o.tnVenta||o.tnVendidas. Pero saveOp() guarda tnVendidas:tnv (las toneladas de compra al proveedor) y nunca guarda tnVenta. Al editar una operación, el campo "Tn a cobrar al comprador" siempre se rellena con el valor de compra.

tn1,tn2,tn3,tn4,tnVendidas:tnv, tn1,tn2,tn3,tn4,tnVendidas:tnv,tnVenta:tnventa,
Ubicación: función saveOp(), objeto data, línea de toneladas

ALTO Bug E — clearForm() no limpia f-tnventa

El campo f-tnventa (Tn a cobrar al comprador) no estaba incluido en la lista de campos a limpiar en clearForm(). Al crear una nueva operación después de una anterior, el campo quedaba con el valor previo.

'f-tnv', /* f-tnventa FALTABA */ 'f-tnv','f-tnventa', // FIX: agregar f-tnventa
Ubicación: función clearForm(), array de IDs de campos

ALTO Bug F — NpTC(): parsing incorrecto de "1.393,50"

La función NpTC() (para leer tipos de cambio) maneja correctamente "1.393" → 1393 y "1.393,50" → 1393.50 en el código original. Sin embargo, si el campo contiene un valor sin coma como "1393.50" (con punto decimal al estilo inglés, posible si el usuario escribe desde un teclado numérico en ciertos dispositivos), el algoritmo stripDotsAlways lo convierte en 139350. Se refuerza la lógica para valores TC >= 100.

else{s=s.replace(/\./g,'');} // strip todos los puntos sin validar else { const stripped = s.replace(/\./g, ''); // TC siempre >= 100 en Argentina; si stripped >= 100, los puntos eran miles s = (parseFloat(stripped) >= 100) ? stripped : s; }
Ubicación: función NpTC(), rama else

MEDIO Bug H — fixEcheqs() nunca se ejecuta

La función fixEcheqs() corrige echeqs con importes en formato de string y elimina duplicados del localStorage. Está definida correctamente pero nunca se llama en el bloque de inicialización al cargar la página.

loadCfg(); fetchDolar(); fixEcheqs(); // FIX: ejecutar al iniciar para limpiar datos del localStorage loadCfg(); fetchDolar();
Ubicación: bloque de inicialización al final del <script>

ALTO Bug I — editOp(): selects de actores se resetean

editOp() llama a fillActorSelects() sin pasar keepVal=true. Esto regenera los <select> de actores perdiendo el valor seleccionado. Aunque los valores se reasignan inmediatamente después con sv()/ss(), hay un momento en que el select queda en valor vacío, y si hay timing issues (por el setTimeout(liveCalc, 50)), puede calcularse con valores incorrectos.

fillActorSelects(); fillActorSelects(true); // FIX: keepVal=true para preservar selección actual
Ubicación: función editOp(), llamada a fillActorSelects
